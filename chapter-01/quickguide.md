# REST APIとMCP - 5分でわかる比較ガイド

**対象:** 非エンジニアのビジネスユーザー、企画担当者、マネージャー

**読了時間:** 5分

---

## 🤔 そもそも何が違うの？

### 一言で言うと

**REST API:** レストランのメニュー（紙）  
**MCP:** スマートなウェイター（AI対応）

---

## 📖 レストランの例えで理解する

### REST API = 紙のメニュー方式

```
あなた（客）
  ↓
紙のメニューを見る
  ↓
「ハンバーグセット1つください」と注文
  ↓
ウェイター（システム）が持ってくる
```

**特徴:**
- ✅ シンプルで分かりやすい
- ✅ 誰でも使える（メニューさえ見れば）
- ❌ メニューに載っていない料理は注文できない
- ❌ 新しいメニューが増えても、紙のメニューは更新されない
- ❌ ウェイターに「今日のおすすめは？」と聞けない

### MCP = スマートウェイター方式

```
あなた（客）
  ↓
ウェイターに話しかける
  ↓
ウェイター「今日のメニューはこちらです」（自動で説明）
  ↓
ウェイター「アレルギーは？」「おすすめは？」（対話的）
  ↓
最適な料理を提案＆提供
```

**特徴:**
- ✅ システム自身が「何ができるか」を説明してくれる
- ✅ AIと対話しながら使える
- ✅ メニューが変わっても自動で対応
- ✅ 「今日のおすすめは？」などの質問に答えられる
- 💡 特にAIアシスタントとの相性が抜群

---

## 📊 実際の使い方の違い

### タスク管理システムの例

#### REST API の場合

**1. タスク一覧を見たい**
```
→ 「GET /tasks というURLにアクセスしてください」
→ http://example.com/tasks を開く
→ タスク一覧が表示される
```

**2. 新しいタスクを作りたい**
```
→ 「POST /tasks というURLに、JSONデータを送ってください」
→ 専用のツール（curlなど）を使って送信
→ タスクが作成される
```

**問題点:**
- どのURLにアクセスすればいいか、**事前に知っておく必要がある**
- ドキュメントを読まないと使えない
- AIに「タスクを作って」と言っても、まずドキュメントを読ませる必要がある

#### MCP の場合

**1. タスク一覧を見たい**
```
AI: 「何ができるか教えて」
→ システム: 「タスク一覧を見れます（task://list）」
→ AI: 「じゃあ見せて」
→ タスク一覧が表示される
```

**2. 新しいタスクを作りたい**
```
AI: 「どんな操作ができる？」
→ システム: 「create_task で新規作成できます。必要なのはタイトルです」
→ AI: 「タイトル『買い物』で作成して」
→ タスクが作成される
```

**メリット:**
- システムに聞けば、**何ができるか自動で教えてくれる**
- ドキュメントを読まなくても使える
- AIが自然に使いこなせる

## 🎯 ビジネス上の違い

### REST API

| 観点 | 評価 | 説明 |
|------|------|------|
| **導入のしやすさ** | ⭐⭐⭐⭐ | 広く使われており、実績が豊富 |
| **開発コスト** | ⭐⭐⭐ | 標準的 |
| **ドキュメント作成** | ⭐⭐ | 別途詳細なドキュメントが必要 |
| **AIとの連携** | ⭐⭐ | 可能だが、事前設定が複雜 |
| **変更への対応** | ⭐⭐ | API変更時にドキュメント更新が必要 |

**向いているケース:**
- Web サイトやモバイルアプリ
- 外部に公開するAPI
- 既存システムとの連携

### MCP

| 観点 | 評価 | 説明 |
|------|------|------|
| **導入のしやすさ** | ⭐⭐⭐ | 新しい技術で学習が必要 |
| **開発コスト** | ⭐⭐⭐ | 初期投資はやや高め |
| **ドキュメント作成** | ⭐⭐⭐⭐⭐ | システム自体が説明するため最小限 |
| **AIとの連携** | ⭐⭐⭐⭐⭐ | AIが自動的に理解して使える |
| **変更への対応** | ⭐⭐⭐⭐⭐ | システムが自己説明するため自動対応 |

**向いているケース:**
- **AIアシスタントとの統合**
- 社内の業務自動化ツール
- エージェントシステム
- 頻繁に機能が変わるシステム

## 💡 具体的なユースケース

### REST API が適している例

✅ **ECサイトのAPI**
- 商品検索、カート、注文など、決まった操作が中心
- 多くのユーザーが同じ方法で使う
- 外部パートナーにも公開する

✅ **SNSのAPI**
- 投稿、いいね、フォローなど、標準的な機能
- 広く使われる必要がある

### MCP が適している例

✅ **AIアシスタント連携**
- 「来週の会議資料を準備して」→ AIが必要なツールを自動選択
- 「先月の売上をまとめて」→ AIがデータを取得・分析

✅ **社内業務の自動化**
- 機能が頻繁に追加・変更される
- 部署ごとに使い方が違う
- AI が文脈を理解して適切に操作する必要がある

✅ **複雑な業務フロー**
- 「承認が必要なタスクを洗い出して、優先度順に並べて」
- AIが複数のツールを組み合わせて実行

## 🤝 併用も可能

**重要:** REST API と MCP は対立するものではありません。

```
外部向けAPI → REST API
     +
AI連携機能 → MCP
```

**実際の例:**
- 一般ユーザー向け: REST API
- Claude などのAIアシスタント向け: MCP
- 両方を提供することで、幅広いユーザーに対応

## ✅ まとめ: 5つのポイント

### 1. 自己説明性
- **REST API:** ドキュメントを読む必要がある
- **MCP:** システムが「何ができるか」を自動で説明

### 2. AI対応
- **REST API:** AIに使わせるには追加設定が必要
- **MCP:** AIがそのまま理解して使える

### 3. 変更への対応
- **REST API:** 変更時にドキュメントも更新が必要
- **MCP:** システムが自動で最新情報を提供

### 4. 使い分け
- **REST API:** 一般的なWebサービス、外部公開API
- **MCP:** AI連携、社内ツール、エージェントシステム

### 5. 将来性
- **REST API:** 実績豊富で安定
- **MCP:** AI時代の新しい標準として期待

## 📞 よくある質問

**Q: REST APIを使っていますが、MCPに移行すべき？**  
A: いいえ、必ずしもその必要はありません。既存のREST APIはそのまま使い続け、AI連携が必要な部分だけMCPを追加することも可能です。

**Q: MCPは難しい技術ですか？**  
A: エンジニアにとっては新しい概念ですが、REST APIと比べて特別に難しいわけではありません。むしろ、自己説明的な設計のため、使う側は楽になります。

**Q: コストはどのくらい違う？**  
A: 初期開発はやや増えますが、ドキュメント作成・メンテナンスのコストが大幅に削減されます。AI連携を考えると、トータルではコスト減の可能性が高いです。

**Q: いつまでに導入を検討すべき？**  
A: AI活用を進めている企業は、今から検証開始をお勧めします。2-3年後には標準的な選択肢の一つになると予想されます。
